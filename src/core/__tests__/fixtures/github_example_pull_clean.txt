This pull request introduces a new ci-feedback.sh script to provide a faster, path-aware local development loop similar to the CI environment. It also refactors the integration test suite to support 'fast' and 'full' profiles, updates documentation, and adjusts server test execution. Several issues were identified in the new scripts and test logic: the git status parsing in the feedback script is fragile and should use git diff, and the script lacks necessary PIP_FLAGS for certain environments. Additionally, the updated assertion logic in api.test.js weakens the test by allowing it to pass on timeouts, and switching server tests to parallel execution by default may introduce flakiness if resources are shared.

Parsing git status --porcelain with awk '{print $2}' is fragile. It fails to handle filenames containing spaces and incorrectly identifies the old path for renamed files (e.g., in R  old -> new, it returns old). Using git diff is a more robust approach for identifying changed files in the working tree.

scripts/ci-feedback.sh
collect_changed_files() {
  local base_ref="$1"

  if [ -n "$base_ref" ]; then
    git diff --name-only "$base_ref"...HEAD
    return
  fi

  git status --porcelain | awk '{print $2}'
}

The git status --porcelain fallback loses rename destinations, but the three-dot diff behavior is intentional.

The script is designed to compare committed changes against the base ref (mimicking GitHub CI behavior) and falls back to working-tree changes only when no base ref exists. The documented behavior confirms this: "If neither ref exists locally..., it falls back to staged + unstaged working-tree changes."

However, the fallback path has a real bug: git status --porcelain | awk '{print $2}' drops rename destinations. For a rename like R  oldname.txt -> newname.txt, it extracts only oldname.txt, missing the new location that tests might depend on.

The fix should focus on the fallback path parsing:

In `@scripts/ci-feedback.sh` around lines 94 - 103, The fallback in
collect_changed_files currently uses `git status --porcelain | awk '{print
$2}'`, which drops rename destinations; update the fallback to use `git status
--porcelain -z` and parse NUL-separated records so renames produce the
destination path (not the source). Specifically, change the parsing in
collect_changed_files to read each NUL-delimited status record, detect rename
entries (status code starting with "R") and emit the second path (the new
filename), and for non-rename entries emit the single path — this ensures rename
destinations are preserved for downstream tests.
